Тема:Рефакторинг исходного кода по принципам SOLID
Цель работы: Изучить практическое применение принципов SOLID, 
научиться проводить рефакторинг существующего решения для приведения его к виду, соответствующему концепциям SOLID.
Задание:
Создайте консольное приложение и подключите к нему проект библиотеки. Выполните рефакторинг приложенной библиотеки классов.

Теоретическое обоснование
Принципы SOLID обеспечивают создание поддерживаемого, расширяемого и тестируемого кода:
SRP (Single Responsibility Principle) — класс должен иметь одну причину для изменения
OCP (Open/Closed Principle) — сущности открыты для расширения, но закрыты для изменения
LSP (Liskov Substitution Principle) — подклассы должны заменять базовые классы без нарушения работы программы
ISP (Interface Segregation Principle) — клиенты не должны зависеть от методов, которые не используют
DIP (Dependency Inversion Principle) — зависимость от абстракций, а не от деталей реализации
Ожидаемый результат: устранение жёсткой связанности, упрощение тестирования, возможность добавления нового функционала без изменения существующего кода.


Описание выполненных действий
1. SRP (OrderProcessor.cs)
Выделены отдельные сервисы: IPaymentProcessorService, IInventoryService, ILoggerService, IReceiptGeneratorService. Класс OrderProcessor теперь отвечает только за координацию обработки заказа.
2. OCP (DiscountCalculator.cs)
Применён паттерн "Стратегия": созданы интерфейсы IDiscountStrategy и IShippingStrategy, реализации вынесены в отдельные классы (RegularDiscount, PremiumDiscount, VipDiscount и др.).
3. LSP (Bank.cs)
Добавлен метод CanWithdraw() для единого контракта проверки, убраны try-catch из класса Bank, унифицирована обработка исключений.
4. ISP (OrderOperations.cs)
Интерфейс IOrderOperations разделён на 6 специализированных: IOrderManager, IPaymentProcessor, IShippingService, IInvoiceGenerator, IReportService, IDatabaseAdmin.
5. DIP (Services.cs)
Создан интерфейс INotificationService, зависимости EmailService и SmsService внедряются через конструктор класса OrderService.
Аналогичные действия выполнены для всех классов библиотеки SOLID_Fundamentals.

Результат выполненной работы
Тестирование показало:
✅ SRP — OrderProcessor корректно обрабатывает заказы, делегируя задачи внешним сервисам
✅ OCP — расчёт скидок работает через стратегии (Regular: 50, Premium: 100, VIP: 150), добавление новых типов не требует изменения существующего кода
✅ LSP — разные типы счетов (SavingsAccount, CheckingAccount) корректно обрабатываются через базовый тип Account
✅ ISP — CustomerPortal реализует только IOrderManager, OrderManager — все необходимые интерфейсы
✅ DIP — зависимости внедрены извне, классы не создают объекты через new
Вывод: Все 5 принципов SOLID успешно применены, код стал более гибким, тестируемым и расширяемым, что соответствует теоретическим ожиданиям.
